# Patch 07 — browser.sw:97

## Problem
`next_msg_id` does a non-atomic read-modify-write on ETS `'next_id`; concurrent `cdp_call` invocations reuse IDs.

## Current code
```sw
fun next_msg_id(session) {
    cur = ets_get(session, 'next_id')
    cur_n = if (cur == nil) { 0 } else { cur }
    nx = cur_n + 1
    ets_put(session, 'next_id', nx)
    nx
}
```

## Proposed fix
```sw
fun next_msg_id(session) {
    random_int(1, 2147483647)
}
```

## Risk
safe

## Confidence
high — replaces the shared counter with a large random integer, eliminating the read-modify-write race while keeping the return type an integer so `wait_for_response` continues to work unchanged.
