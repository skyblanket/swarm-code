# parallel test-gen experiment · 2026-05-24

24 `swarm-code -p` headless subagents, one per `src/*.sw` module.
Prompt: read the module, identify pure exported functions, write a
self-contained `module Main` test file under `tests/gen/<module>_test.sw`.

## Tally

- **24/24** files generated  (~7 min wall, ~$0.80 in Kimi tokens)
- **8/24** built + ran cleanly → kept in `tests/` (33% real coverage)
- **2/24** parse errors — invalid sw syntax (9% hallucination rate)
- **14/24** build-failed due to transitive-import limit in swc — the
  test code itself is correct (calls real exports, real signatures)
  but `swc build test.sw` doesn't follow LLM/Memory/Mcp's deeper
  imports. Archived under `docs/internal/failed-tests/` for follow-up.

## Passing tests

`tests/agent_test.sw`, `browser_test`, `prompts_test`, `reader_test`,
`scheduler_test`, `telemetry_test`, `trajectory_test`, `util_test`.

Run with `./tests/run_all.sh` (or `make` target later).

## Two bugs surfaced about swarm-code itself

1. Parallel `-p` subagents all resume the same `.active` session journal
   by default. With 163 messages in the active session, 12 of the 24
   swarms auto-compacted on startup and got an empty Kimi reply.
   Workaround in the launcher: `rm .active` before each job. Better
   fix: `--no-resume` flag (TODO).

2. `swc build` on a test file that imports a deep-dep module
   (e.g. `import LLM` where LLM imports `Util`/`ToolSchemas`) doesn't
   transitively pull in those deps, producing
   `undefined LLM_chat_completions_url` link errors. The tests are
   semantically correct.

## Honest verdict

For mechanical coverage seeding: 1/3 hit rate from a $1 parallel run
is decent. The signal: parallel swarm dispatch is real engineering
leverage when the task is **narrow** + **independent per worker** +
**verifiable end-to-end** (build + run).
